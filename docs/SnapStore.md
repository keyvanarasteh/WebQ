# Publishing WebQ to the Snap Store

To publish the **WebQ** (Tauri + Svelte 5 + Rust) application to the Snap Store in Linux, you'll need to package it using `snapcraft`. Since Tauri relies heavily on system libraries like GTK and WebKitGTK, Snapcraft handles these dependencies effortlessly using the `gnome` extension.

Here is the step-by-step guide to configure, build, and publish WebQ to the Snap Store.

## 1. Prerequisites
First, you'll need a developer account on Snapcraft and the tools installed on your local Linux machine to test the build.
* **Install snapcraft & lxd/multipass**:
  ```bash
  sudo snap install snapcraft --classic
  ```
* **Register Developer Account**: Go to [snapcraft.io](https://snapcraft.io/) and create an Ubuntu One account. Register the name `webq` (or your preferred unique package name) in your developer dashboard.

## 2. Create the `snapcraft.yaml` Configuration
In the root directory of your project, create a `snap` directory and a `snapcraft.yaml` file:

```bash
mkdir snap
touch snap/snapcraft.yaml
```

Populate `snapcraft.yaml` with the following configuration. This configuration utilizes the **npm** and **rust** plugins to build the Svelte frontend and Tauri backend correctly within the isolated snap container:

```yaml
name: webq # Must match the exact name you registered on Snapcraft.io
base: core22 # Ubuntu 22.04 base
version: '1.0.0'
summary: A high-performance web intelligence and cybersecurity platform.
description: |
  WebQ is a cross-platform Web Intelligence and Cybersecurity platform 
  built with Rust, Svelte 5, and Tauri, powered by the web-analyzer crate.

grade: stable # 'devel' for testing, 'stable' for releases
confinement: strict # Strict isolation is required for the Snap Store

apps:
  webq:
    # binary generated by Cargo
    command: usr/bin/webq
    # Gives the app access to essential X11/Wayland/GTK interfaces
    extensions: [gnome] 
    plugs:
      - network
      - network-bind
      - home

parts:
  node-deps:
    plugin: npm
    source: .
    npm-node-version: "20.10.0"

  webq-gui:
    after: [node-deps]
    plugin: rust
    source: src-tauri
    # Make sure Cargo builds the release binaries into the Snap hierarchy
    override-build: |
      npm run tauri build
      mkdir -p $CRAFT_PART_INSTALL/usr/bin
      cp target/release/webq $CRAFT_PART_INSTALL/usr/bin/webq
    build-packages:
      - libwebkit2gtk-4.1-dev
      - build-essential
      - curl
      - wget
      - file
      - libxdo-dev
      - libssl-dev
      - libayatana-appindicator3-dev
      - librsvg2-dev
```

> **Note**: Depending on your `tauri.conf.json`, you might need to adjust the binary name (`webq`) under `apps` and `override-build`. Also ensure `build-packages` includes any native C-dependencies needed by your core crates (e.g., `nmap`, etc. if used by `web-analyzer`).

## 3. Build and Test Locally
Before pushing to the store, build the `.snap` package locally to ensure it works. 
Run the following command at the root of your project:
```bash
snapcraft
```
This will spin up an isolated LXD container or Multipass VM, install the dependencies, build the Svelte app, compile the Rust backend, and output a `webq_1.0.0_amd64.snap` file directly in your project root.

To test the generated installer on your machine:
```bash
sudo snap install ./webq_1.0.0_amd64.snap --dangerous
```
*(The `--dangerous` flag is needed because the package isn't signed by the official Snap Store yet).*

## 4. Publish to the Snap Store
Once the snap installs and runs flawlessly on your local environment, you can publish it:

1. **Login to Snapcraft CLI**:
   ```bash
   snapcraft login
   ```
2. **Upload the package**: Push it to the `edge` channel first so you can soft-test the distribution:
   ```bash
   snapcraft upload --release=edge webq_1.0.0_amd64.snap
   ```
3. **Promote to Stable**: Once you are satisfied with the `edge` release, you can promote it to `stable` via the web dashboard at `snapcraft.io`, or directly using the CLI:
   ```bash
   snapcraft release webq <revision-number> stable
   ```

## 🎯 CI/CD Automation (Recommended)
Since the project contains a `.github` folder, the best-practice method is to automate this via GitHub Actions so you don't continually compile Snaps on your local machine.

You can configure Snapcraft's automated build service directly via your web dashboard ([Snapcraft Builds](https://snapcraft.io/builds)) by connecting your GitHub repository. Whenever you push to the `main` branch, Snapcraft will automatically trigger a build using your `snapcraft.yaml` and publish it straight to the `edge` channel.
